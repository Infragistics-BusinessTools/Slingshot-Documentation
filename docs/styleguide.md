# Slingshot Documentation Style Guide

## 1. Purpose

This style guide defines writing standards for Slingshot product documentation to ensure a consistent, clear, and user-focused experience. It reflects the tone and structure used throughout the existing Slingshot Help Center.

Our goal is to create documentation that is:
- Easy to understand  
- Actionable  
- Consistent in structure and voice  

---

## 2. Tone and Voice

### Core Principles

- **Clear and direct** – Use straightforward language.
- **User-focused** – Write for end users, not internal teams.
- **Professional but approachable** – Friendly without being casual.
- **Helpful and supportive** – Guide users toward success.

### Avoid

- Overly technical jargon without explanation  
- Marketing-heavy or promotional language  
- Humor, slang, or conversational filler  

---

## 3. Headings and Structure

Use a clear hierarchy of headings.

```
# Page Title
## Major Section
### Subsection
#### Minor Subsection (if needed)
```

### Heading Style

- Use **Title Case** (capitalize main words)
- Keep headings concise and descriptive
- Do not use punctuation at the end of headings

**Good**
```
## Create a Dashboard
```

**Avoid**
```
## Creating a dashboard.
```

---

## 4. Paragraph Style

- Keep paragraphs short (1–4 sentences)
- Focus each paragraph on a single idea
- Start with the most important information

**Example**
```
Dashboards help you visualize your data in one place. You can combine charts, metrics, and filters to monitor performance.
```

---

## 5. Lists

### Bullet Lists

Use bullets when order does not matter.

```
* Dashboards
* Projects
* Goals
```

### Numbered Lists

Use numbered steps for procedures or sequences.

```
1. Open the workspace.
2. Select **New Dashboard**.
3. Add your first widget.
```

---

## 6. Writing Instructions

When describing actions:

- Use **imperative verbs**
- Be direct and specific
- Write steps in the order they happen

**Good**
```
Click **Create Project**.
```

**Avoid**
```
You will need to click the Create Project button.
```

---

## 7. Feature Descriptions

When introducing a feature, follow this structure:

1. **What it is**
2. **Why it matters**
3. **How it helps the user**

**Example**
```
Goals help teams align their work with measurable outcomes. By tracking progress in one place, teams can stay focused and accountable.
```

---

## 8. Images and Visuals

- Place images directly after the paragraph that introduces them
- Use images only when they add clarity
- Include descriptive alt text

**Example**
```
![Dashboard overview showing widgets and filters](dashboard-overview.png)
```

Avoid decorative or redundant images.

---

## 9. Callouts and Notes

Use callouts to highlight important information such as limitations, requirements, or tips.

```
> [!NOTE]
> You must have Editor permissions to create dashboards.
```

```
> [!TIP]
> Use filters to focus on specific teams or time periods.
```

### Guidelines

- Keep callouts short
- Do not overuse them
- Only include truly important information

---

## 10. Language and Grammar

- Use **American English**
- Prefer **active voice**

**Good**
```
Open the Analytics view.
```

**Avoid**
```
The Analytics view can be opened.
```

- Avoid unnecessary filler words
- Use present tense whenever possible

---

## 11. Technical Content (APIs, Settings, Integrations)

For technical documentation:

- Define technical terms the first time they appear
- Include examples where helpful
- Use code blocks for commands, JSON, or API samples

**Example**
```
### API Overview

The Slingshot API uses REST principles. Requests return data in JSON format.
```

---

## 12. Supported Languages Section

When listing supported languages:

```
## Supported Languages

* Chinese (Simplified)
* Chinese (Traditional)
* Dutch
* English
```

If applicable, include a short note about fallback behavior when a language is not supported.

---

## 13. Footer and Legal

When required, include standard legal and support links at the bottom of documentation pages.

```
* Privacy Policy  
* Cookies  
* Terms of Use  
* License Agreement  
```

Include the official copyright statement where appropriate.

---

## 14. Standard Page Template

Use this template when creating new documentation pages.

```
# Page Title

Brief introduction explaining what the article covers and why it matters.

## What Is This?

Explain the feature, tool, or concept.

## How to Use It

1. Step one
2. Step two
3. Step three

> [!NOTE]
> Important information the user should know.

## Tips and Best Practices

Helpful guidance to improve results.

## Related Articles

* Link to related article
* Link to related article
```

---

## Goal

All documentation should be **clear, structured, helpful, and consistent** with the Slingshot Help Center experience. When in doubt, prioritize clarity and usefulness over completeness or technical depth.
